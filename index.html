Business Problem

Non-technical users couldn't get executive-level insight from raw datasets without a data analyst sitting between them and the answer — a bottleneck that slowed decision-making by days.

Engineering Solution
  • Multi-agent workflow — 7 specialized agents running in parallel
  • DuckDB over hosted warehouse — in-process analytical queries with < 2.1s latency
  • Qdrant vector store — semantic search over past analyses
Business Impact
50+ Users
2.1s Avg insight
0 Analyst bottlenecks
Days→Min Insight time
System Architecture
  • Ingestion: CSV/JSON upload with DuckDB schema inference
  • Orchestration: LangGraph-based agent coordinator with state management
  • Agents: 7 specialized agents (planner, statistician, root-cause, forecaster, RAG, synthesizer, validator)
  • Storage: DuckDB for analytics, Qdrant for vector embeddings
  • API: FastAPI with JWT and role-based access
Data Flow

User upload → schema inference → natural language parsing → parallel agent execution → report synthesis → executive summary with visualizations.

Each agent operates independently with shared state via LangGraph's checkpointing system, enabling fault isolation and individual iteration.

Architecture Decisions
  • Agent granularity: 7 specialized agents vs. one monolithic — chosen for debuggability, independent iteration, and fault isolation
  • DuckDB trade-off: Accepted per-query compute cost vs. provisioned warehouse overhead — won for low concurrency, high-variety workloads
  • JWT + RBAC: Auth as a system property, scoped to dataset-level permissions
  • Qdrant: Chosen for balance of performance, ease of deployment, and semantic search
Security & Reliability
  • JWT authentication with refresh tokens
  • Role-based access control (admin, analyst, viewer)
  • Rate limiting per user to prevent abuse
  • Structured logging with request tracing
  • Graceful degradation — if one agent fails, others continue

Business Problem

Generic travel sites surface popular destinations based on star ratings alone, ignoring the nuance of review sentiment and individual traveler preferences.

Engineering Solution
  • Content-based filtering + BERT sentiment — nuanced sentiment from review text
  • Weighted scoring — popularity and sentiment weighted separately
  • K-NN similarity — attribute proximity ranking
Business Impact
40% Higher engagement
4.5 /5 satisfaction
~180ms Inference
1:1 Personalized
System Architecture
  • Preference Engine: User input parsing for budget, style, and interests
  • Filtering Layer: Attribute-based candidate reduction
  • Sentiment Layer: BERT-based sentiment extraction with caching
  • Ranking Layer: K-NN similarity with weighted scoring
  • Visualization: Plotly interactive charts with destination comparison
Data Flow

User preferences → attribute filtering → BERT sentiment extraction → K-NN ranking → personalized recommendations with visualizations.

Embeddings are cached per destination, reducing inference cost by 60% for repeat queries.

Architecture Decisions
  • BERT over lighter NLP: Deeper understanding of review nuance, accepted ~180ms inference
  • Flask over FastAPI: Workload didn't need async — simplicity won the trade-off
  • Embedding cache: Pre-computed embeddings for repeat destinations, 60% cost reduction
  • Weighted scoring: Separated popularity from sentiment to avoid polarizing results
Security & Reliability
  • Input sanitization and validation
  • Graceful fallback to popularity-based recommendations
  • Embedding cache with TTL for freshness
  • No PII stored — fully stateless
